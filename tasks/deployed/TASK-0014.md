## Problem

TASK-0013 resolved the primary ANOTE-web demo report-generation latency regression by changing the `anote-web-api` Container App model deployment from `gpt-5-mini` to `gpt-4-1-mini` with `gpt-5-nano` fallback. That restored direct warm `/report` latency from roughly 19 s to 4.7-6.3 s, but the remaining ANOTE-web hardening work was explicitly deferred.

The demo still has unresolved web-side reliability, responsiveness, and transcription-quality risks across Azure config, the ANOTE-web FastAPI backend, the SWA proxy layer, and the browser client.

## Desired Outcome

- ANOTE-web demo report generation remains inside the app-3 boundary and is hardened for latency, cold-start behavior, streaming responsiveness, and audio quality.
- Azure configuration changes are limited to the ANOTE-web deployment surfaces: `anote-web-api` Container App and ANOTE-web Static Web App configuration.
- Browser and proxy changes are limited to the ANOTE-web repository.
- `ANOTE_mobile` is used only as read-only architectural reference where explicitly noted; no mobile code, config, paths, or deployment changes are introduced.
- Progress is trackable by the sub-items below and can be shipped in separate PRs in the requested order.

## Scope Breakdown

- [ ] **A-1 Chat model deployment (Azure config, `anote-web-api`)** — evaluate `gpt-4o-mini` versus current `gpt-4-1-mini` on Czech reports of about 4 k chars; adopt the faster option using env-only `az containerapp update`; success metric: `/report` p50 <= 6 s.
- [ ] **A-2 `minReplicas = 1` on `anote-web-api`** — env-only or scale-config-only Azure change to eliminate the SWA cold-start 500 path.
- [ ] **B-3 SSE streaming for `/report`** — in ANOTE-web's own FastAPI backend (`anote-web-api`), add `text/event-stream` output gated by a query flag; pipe streaming through `route.ts`; rewrite `streamReport` in `useDemoSession.ts` so `onToken` is called incrementally instead of once with the full text.
- [ ] **B-4 Debounce periodic `/report`** — in `useDemoSession.ts`, skip regeneration when the transcript grew by fewer than 30 characters since the last request.
- [ ] **B-5 Cancel in-flight `/report` on newer chunk** — use the existing `AbortController`; always service the latest transcript and drop stale requests.
- [ ] **C-6 Client-side VAD** — integrate `@ricky0123/vad-web` with Silero ONNX via `onnxruntime-web` inside `useMediaRecorder.ts`; upload only chunks with detected speech; if VAD init fails, fall back to current raw-upload behavior. Replicate the pattern inside ANOTE-web from scratch; do not import from or edit `ANOTE_mobile`.
- [ ] **C-8 Whisper initial-prompt anchor** — in `route.ts`, add a short Czech medical prompt multipart field such as `Leka\u0159sk\u00e1 zpr\u00e1va. Pacient ud\u00e1v\u00e1...` to the Azure Whisper request.
- [ ] **D-10 Transcription model upgrade (Azure + SWA config)** — switch `AZURE_WHISPER_ENDPOINT` to `gpt-4o-mini-transcribe` on `anote-openai-swe` (Sweden Central); add `anote-openai-swe.openai.azure.com` to the `connect-src` CSP whitelist in `staticwebapp.config.json`; verify Czech accuracy with a small independent smoke sample that does not reuse the mobile FLEURS set.
- [ ] **E-11 Pre-warm ping** — in `page.tsx`, fire a non-blocking `GET /api/demo/health` on demo page mount and add the small proxy route.
- [ ] **E-12 Client retry** — in `streamReport` within `useDemoSession.ts`, retry once on 5xx before surfacing an error.

## Reasoning

This is a non-trivial follow-up task rather than a direct extension of TASK-0013 because the remaining work spans multiple delivery surfaces: Azure deployment config, SWA proxy behavior, browser hooks, and the ANOTE-web-specific backend flow. The current ANOTE-web codebase already shows clear touchpoints for pre-warm, retry, streaming, and transcription changes, so the work is actionable inside app-3 without broad cross-app discovery.

It is still too large for one implementation task. The scope bundles five shippable workstreams with different risk profiles: Azure-only config changes, small client/proxy latency mitigations, transcription-quality hardening, report streaming/state-management changes, and endpoint/CSP migration validation. A lightweight spec is the cleanest next step because it can preserve the requested order while defining the child-task boundaries and the interface expectations for the higher-risk streaming and VAD work.

## Split Recommendation

Recommended structure: keep `TASK-0014` as a parent task with ordered follow-up execution tasks.

1. Config hardening: A-1 and A-2
2. Cold-start mitigation: E-11 and E-12
3. Transcription quality hardening: C-6 and C-8
4. Report streaming and request control: B-3, B-4, and B-5
5. Transcription endpoint/CSP migration and smoke validation: D-10

This split preserves the original grouping and sequencing intent while keeping each execution task coherent enough to ship and validate independently.
