## Problem

The ANOTE-web demo page was failing at the report-generation step. During the reported SWA demo failure, the user-facing symptom was a 5xx backend call failure surfaced by the SWA gateway instead of a generated Czech medical report.

## Desired Outcome

- Report generation works again on the ANOTE-web demo page.
- The ANOTE-web implementation continues to use its own web backend flow and repository code.
- Investigation compares the mobile reference behavior against ANOTE-web only as needed to identify the web-side cause.
- Any implementation or configuration changes remain strictly inside the ANOTE-web web-backend deployment boundary.
- The fix scope stays limited to what is needed to restore report generation reliability.

## Investigation Findings (confirmed)

### Root cause

The primary cause was **not an ANOTE-web application-code defect**. The `anote-web-api` Azure Container App in resource group `anote-rg` (West Europe) was configured with:

- `AZURE_OPENAI_DEPLOYMENT=gpt-5-mini`

That reasoning model produced materially slower report-generation latency than the mobile reference backend.

The mobile reference backend `anote-api` in resource group `ANOTE` (West Europe) uses the same FastAPI source and the same `anote-openai` Azure OpenAI resource, but it was configured with:

- `AZURE_OPENAI_DEPLOYMENT=gpt-4-1-mini`
- `AZURE_OPENAI_FALLBACK_DEPLOYMENT=gpt-5-nano`

Using an identical Czech `/report` prompt, measured latency was:

- mobile backend: **6.3 s**
- web backend before fix: **19.2 s**

The reported SWA demo failure was therefore the result of **SWA cold-start plus the web backend's slow upstream generation call exceeding the SWA function response window**, which surfaced to the demo as a backend 5xx failure.

### Boundaries honored

- `ANOTE_mobile` was inspected in read-only mode only.
- No mobile code or mobile configuration was changed.
- The ANOTE-web backend architecture was preserved.
- The remediation was limited to aligning the web backend model deployment environment with the mobile reference behavior.

## Fix Applied

On **2026-04-23**, the `anote-web-api` Container App environment was updated in Azure using:

```bash
az containerapp update -n anote-web-api -g anote-rg --set-env-vars AZURE_OPENAI_DEPLOYMENT=gpt-4-1-mini AZURE_OPENAI_FALLBACK_DEPLOYMENT=gpt-5-nano
```

This was an **infrastructure-only fix**. No ANOTE-web repository code change was required.

## Validation

- `az containerapp show` confirmed the updated environment variables on `anote-web-api`.
- Direct `/report` `curl` calls against the web Container App returned `200` with well-formed Czech reports.
- Post-fix latency across three warm `/report` calls was **4.7-6.3 s**.
- Pre-fix reference latency for the same web backend path was **19.2 s**.
- The post-fix latency now matches the mobile reference backend range closely enough to remove the previously observed model-selection gap.

## Remaining Validation Boundary

End-to-end verification through the Azure Static Web Apps gateway under cold-start conditions has **not** been re-captured in this control-layer task file.

Current expectation: the prior 500s should stop occurring because the upstream `/report` call now completes comfortably within the SWA response window. This expectation is strongly supported by the measured latency drop, but it is still an inference until a fresh cold-start SWA path check is recorded.
