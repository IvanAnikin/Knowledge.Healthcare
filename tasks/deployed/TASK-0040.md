## Problem

### Confirmed Facts

- The raw task request targets **app-3 (ANOTE-web)**.
- The raw request states that in the **mobile version**, when the user is **not at the very top of the page**, the navbar has **no background**.
- The raw request states that navbar text/menu options then **overlap with page content** and are **hard to read**.
- The raw request states that when the user clicks the **hamburger button** in that scrolled mobile state, the menu should open with the **same background behavior** it already has when opened while the user is at the top of the page.
- `applications/index.md` confirms `app-3` maps to **ANOTE-web**.
- `applications/app-3-overview.md` confirms ANOTE-web contains layout components under `src/components/layout/`, including a navbar area, and global styling in `src/app/globals.css`.

### Inferred Likely Impact

- Likely: this is a **mobile navigation readability bug** on the public marketing site.
- Likely: the defect affects usability and may reduce clarity of primary navigation/CTA access on phones.

## Desired Outcome

### Confirmed Facts

- On mobile, when the user opens the hamburger menu while scrolled away from the top, the navbar/menu uses a readable background.
- The applied background behavior should match the existing behavior already seen when the same menu is opened while the user is at the top of the page.
- Menu options should remain clearly legible instead of visually blending into page content.

### Inferred Likely Impact

- Likely: the fix should behave consistently across the main localized routes, not only on one language variant.

## Constraints

### Confirmed Facts

- Scope is limited to **app-3 / ANOTE-web**.
- This is a UI bug report, not a request for a redesign.

### Recommendations

- Reuse the current top-of-page open-menu background treatment rather than introducing a new mobile navbar visual pattern.
- Validate in at least one mobile viewport before promoting beyond `done`.
