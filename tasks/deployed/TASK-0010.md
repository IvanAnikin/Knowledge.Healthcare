# Task: Investigate and Fix CSP eval Warning (script-src blocked)

## Problem

The browser reports a Content Security Policy (CSP) violation for the `script-src` directive, blocking eval-style JavaScript evaluation.

## Desired Outcome

- The root cause of the eval call at runtime is identified.
- The CSP is tightened by removing `'unsafe-eval'` from `script-src` if it is not genuinely required.
- If removal is not possible without breaking functionality, the source of the eval call is documented and the decision to retain `'unsafe-eval'` is recorded as a decision entry.

## Investigation Findings (confirmed)

### CSP is configured in hosting config, not Next.js

**File:** `staticwebapp.config.json` (Azure Static Web Apps hosting config)

Current `script-src` directive:
```
script-src 'self' 'unsafe-inline' 'unsafe-eval' https://plausible.io
```

`'unsafe-eval'` is **explicitly present**. `next.config.ts` has no `headers()` function and no CSP config — the CSP is set entirely at the hosting layer.

### Sources investigated

| Candidate | Finding | Confirmed eval? |
|-----------|---------|-----------------|
| Application source (`src/`) | Zero `eval()`, `new Function()`, `setTimeout(string)`, `setInterval(string)` patterns | No |
| `framer-motion` v12.38.0 | No eval found in `framer-motion.dev.js` | No |
| Plausible `script.js` | Standard lightweight tracker; no eval | No |
| `lottie-web` (via `lottie-react`) | **`eval()` confirmed** at `node_modules/lottie-web/build/player/lottie.js:14422` for AE expressions | Yes — but dormant |
| Next.js dev mode (webpack/Turbopack) | Uses eval-style source maps in development | **Dev-only** |

### lottie-web eval — dormant but present

`lottie-web` uses `eval()` to execute After Effects expression strings (Bodymovin). However, `lottie-react` is installed (`package.json`) but **not imported anywhere in `src/`**. The eval code path is only reached when a Lottie animation with AE expressions is actually rendered. Currently dormant.

### Most likely reason `'unsafe-eval'` was added

Either: (a) Next.js dev mode triggered the warning during development and `'unsafe-eval'` was added as a quick fix without distinguishing dev vs. production, or (b) it was added preemptively for `lottie-react` which is not yet used.

### Production vs. dev relevance

- In **development**, Next.js (webpack/Turbopack) uses `eval` for source maps. This is the most likely original trigger. It is **not a production concern**.
- In **production**, no application-level eval call exists. The browser warning in production — if it occurs — would most likely come from a future use of `lottie-react` with AE expressions, or from a dependency not yet investigated.

## Recommended Fix Direction

1. **Remove `'unsafe-eval'` from `staticwebapp.config.json`** as a first step. Test in production (or staging) to confirm no runtime errors.
2. **Remove `lottie-react`** from `package.json` if it is confirmed unused — this eliminates the dormant eval source and reduces bundle size.
3. **If removing `'unsafe-eval'` breaks anything in production**, capture the exact CSP violation error (source file + line number from browser console) and document it in this task before deciding on a fix.
4. Do **not** add `'unsafe-eval'` back without a documented reason and decision record.

## Suspected Affected Files

- `staticwebapp.config.json` — remove `'unsafe-eval'` from `script-src`
- `package.json` — remove `lottie-react` if unused (verify first)
- `package-lock.json` / `node_modules` — updated after package removal

## Unknowns

- Whether `'unsafe-eval'` removal causes any runtime error in production (requires a test deployment to confirm).
- Whether `lottie-react` is planned to be used in future features — confirm with product owner before removing.
