# Task: Include transcript alongside report in outgoing email

## Metadata

- **ID:** TASK-0037
- **Created:** 2026-05-02
- **Author:** User (via @task-triage)
- **Status:** active (ready for execution — spec approved 2026-05-02)
- **Spec:** [`specs/app-2/SPEC-0037-transcript-in-email.md`](../../specs/app-2/SPEC-0037-transcript-in-email.md)
- **App(s):** app-2
- **Priority:** medium
- **Type:** feature

## Problem

When a session/visit email is sent to the user from ANOTE_mobile, only the generated medical report is currently included in the email content. The full transcript that produced the report is not included, so the recipient cannot review the source dictation alongside the structured report in a single message.

## Desired Outcome

A single outgoing email contains BOTH the transcript and the report:

- Transcript appears first.
- Report appears directly beneath the transcript.
- Existing report formatting and delivery continue to work.
- Behavior is verified end-to-end: the email is sent and the received message contains both sections in the correct order.

## Constraints

- **Single email** — must not be split into two separate messages.
- **Order is fixed** — transcript first, then report.
- **PHI / privacy (GDPR-relevant):** transcript is high-sensitivity Czech clinical data (per `applications/app-2-overview.md` data table). Whatever delivery mechanism is chosen must remain consistent with the existing GDPR posture (transcripts are already transmitted via HTTPS to the backend; SMTP delivery is an existing optional feature, see `backend/main.py` `/send-report-email`). HIPAA is not the governing regime for this app (Czech / EU users), but treat email content as sensitive clinical data either way.
- **Existing optional/disabled-by-default email pathway** — the SMTP feature is currently disabled unless `SMTP_HOST` is set; this change must not alter that gating.
- Must preserve the existing report rendering / formatting.

## Suspected Affected Areas

Resolved during spec investigation (see spec for full detail):

- `backend/main.py` — `SendReportEmailRequest` model + `send_report_email` body assembly.
- `backend/tests/test_email_endpoint.py` — add body-composition unit test.
- `mobile/lib/services/report_service.dart` — add `transcript` to `sendReportEmail` payload.
- `mobile/lib/providers/session_provider.dart` — pass `state.transcript` from `_sendEmailIfEnabled` and `sendReportEmailNow`.

## Decisions (locked 2026-05-02)

All open questions from triage and from the spec's first draft have been resolved by the user:

1. **Delivery format:** inline plain-text — both transcript and report in the same body. No attachments, no HTML, no multipart MIME.
2. **Size threshold:** none. v1 is fully inline regardless of transcript length. No attachment fallback.
3. **Privacy / data-flow:** unchanged. Mobile sends the transcript explicitly in the `/send-report-email` payload via an additive optional `transcript` field on `SendReportEmailRequest`. Backend logging policy unchanged (transcript and report content not logged).
4. **Email-sending location:** confirmed backend-side via FastAPI `/send-report-email` + SMTP.
5. **Headings & separator:** confirmed `--- Přepis ---` and `--- Lékařská zpráva ---`, reusing the existing `---` separator vocabulary already in the email body.
6. **Internationalization:** Czech, consistent with the rest of the app.
7. **Backwards compatibility:** preserved by making `transcript` optional with `""` default. `SendReportEmailRequest` keeps Pydantic's permissive default for unknown fields (no `extra="forbid"` introduced in this task).

## Acceptance Criteria

Mirrored from the spec (full list lives in `specs/app-2/SPEC-0037-transcript-in-email.md` §Validation):

- A single email is sent (not two separate emails).
- Transcript appears first under heading `--- Přepis ---`; report appears below under heading `--- Lékařská zpráva ---`. Both inline plain-text in the same body.
- Everything is inline regardless of transcript length (no size cap, no attachment fallback).
- `SendReportEmailRequest` retains permissive default for unknown fields (no `extra="forbid"`).
- Existing `test_email_endpoint.py` continues to pass; new body-composition test asserts the layout above.
- End-to-end manual verification with `SMTP_HOST` configured produces the expected received email.

## Notes

- Spec-writer investigation confirmed the `/send-report-email` endpoint payload does NOT carry the transcript today, so the change is cross-layer (backend + mobile), still single-app (app-2 only).
- Scope verified at 4 files; not cross-app.
