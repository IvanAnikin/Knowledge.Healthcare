## Problem

`Health-Analysis` does not currently expose a UI language switch, and Czech is not a supported UI language. The product audience includes Czech-speaking clinicians (consistent with the broader project context: app-1 medical_advisor is a Czech diabetes education chatbot, and app-3 ANOTE-web is bilingual cs/en). The Health-Analysis app needs (a) a localization framework in place and (b) Czech as a selectable language alongside the current default.

## Desired Outcome

- A UI language switcher is available somewhere in the Health-Analysis app shell.
- Czech (`cs` / `cs-CZ`) is a fully supported UI language for at least the in-scope strings (definition of scope is itself an open question — see Unknowns).
- The previously default UI language remains supported and selectable.
- Locale selection persists across sessions in a defined way (per-user setting and/or browser preference — see Unknowns).
- Date, number, and unit formatting respect the active locale where reasonable, with explicit handling of clinical units (e.g. `mmol/L`).

## Confirmed Facts

- The user requested: "Add a language switch adding czech language."
- The target application "health-analysis" is registered in `applications/index.md` as **app-4 (Health-Analysis)**.
- The Health-Analysis web app is built with Next.js + React (per `TASK-0021` and `TASK-0029`).
- `TASK-0029` established a shared visual system in `web/app/globals.css`, sticky glass app shell and persisted theme behavior in `layout.tsx`, and a `ThemeToggle.tsx` — providing a precedent for a similarly placed locale toggle.
- The user has provided no separate translation file or glossary for Czech UI strings as part of this request.

## Assumptions (flagged, not confirmed)

- **Assumption:** The current default UI language of Health-Analysis is English. (Needs confirmation by inspecting the running UI / source. If the current copy is Czech-only, the "language switch" may be the inverse direction — adding English.)
- **Assumption:** No i18n framework (e.g. `next-intl`, `next-i18next`, `react-intl`) is wired into Health-Analysis yet. (Needs confirmation.)
- **Likely:** Translatable strings are currently inline in TSX components rather than centralized in dictionary files, so part of the work is extraction.
- **Likely:** Clinical content (e.g. assumption lines, heuristic warnings, validation messages) is currently authored in one language and will need a translation pass alongside structural i18n changes.

## Constraints

- Must not regress the deterministic-analytics-first safety framing or alter the meaning of clinical assumption/warning text in either language.
- Unit symbols that are internationally standard (e.g. `mmol/L`, `mg/dL`) must remain unchanged across locales.
- Theme persistence pattern from `TASK-0029` should be a reference point; locale persistence should be similarly minimal and predictable.
- Reduced-motion / accessibility behavior must not regress.

## Suspected Affected Areas

- `web/app/layout.tsx` — locale provider wiring, lang attribute on `<html>`, switcher placement in the app shell.
- `web/components/` — likely a new `LanguageSwitch.tsx` modeled on `ThemeToggle.tsx`.
- All user-facing TSX in `web/app/` and `web/components/` — extraction of inline strings to dictionaries.
- New `web/messages/` (or `web/locales/`) directory with `en.json` / `cs.json` (path depends on chosen i18n library).
- `web/lib/api.ts` types — only if backend-returned strings (validation messages, assumption lines, heuristic copy) need a localization contract.
- `api/app/...` — possibly affected if validation/analytics messages currently embed user-facing strings; ideally backend returns stable codes/keys and frontend renders them.

## Unknowns / Open Questions

- Is there already any i18n infrastructure in Health-Analysis (any dictionary files, `next-intl`, `next-i18next`, or per-component locale logic)?
- Scope of strings to translate:
  - UI chrome only (navigation, buttons, panel titles)?
  - Plus deterministic-analytics labels (chart titles, table headers)?
  - Plus assumption/heuristic/warning text?
  - Plus LLM summary output (does the LLM provider need a `language` instruction, and does the audit log capture which language was requested)?
- Translation source/process: who provides the Czech copy — product owner, professional translator, reuse of strings already established in app-3 (ANOTE-web) or app-1 (medical_advisor)?
- Locale persistence:
  - per-user (server-side, requires user identity) vs.
  - per-browser (cookie / `localStorage`) vs.
  - URL-prefixed routing (e.g. `/cs/...`, `/en/...`) as in app-3.
- Default locale and fallback chain.
- Date / number / time-of-day formatting: use `Intl` directly, the i18n library's formatter, or a separate utility?
- Whether `cs-CZ` should also imply 24-hour time, comma decimal separator, and Czech-style date order across all charts and tables (yes is the intuitive answer, but should be confirmed).
- Cross-app reuse: are there shared UI components or a shared design system between `Health-Analysis` and other apps that would make this a cross-app concern? Current evidence (separate repos, no shared component package documented) suggests **no**, but worth confirming.
- Interaction with `TASK-0033` (Broz CSV automatic mapping to Czech-localized field names): if mapping target labels are the canonical Czech UI labels, the mapping table is effectively part of the i18n vocabulary.

## Notes

- Closely related to `TASK-0033` (Broz CSV automatic mapping to Czech-localized field names). The two tasks share vocabulary and should be coordinated so that field names used in the mapping table match the i18n dictionary entries used in the UI.
- App-3 (ANOTE-web) already implements a cs/en bilingual experience via a `[lang]` URL prefix; that pattern is a candidate reference, though the fit for Health-Analysis (mostly a single authenticated workflow) may be different.

## Reasoning

Impact is **high** because Czech is the primary user-facing language across this project's clinician audience and is a likely prerequisite for any pilot use of Health-Analysis with Czech doctors. This warrants a spec — the work touches every user-facing surface and a wrong framework choice is expensive to undo.

## Cross-App Considerations

No direct cross-app code coupling is known (Health-Analysis has its own repo and stack). However, vocabulary alignment with app-3 (ANOTE-web) Czech copy and any clinical terminology used in app-1 (medical_advisor) would improve consistency for clinicians who use multiple apps. This is a soft coordination concern, not a hard cross-app dependency, so the task remains tagged `app-4`.
