## Problem

When a user uploads a "Broz" CSV into `Health-Analysis`, the source columns currently require manual mapping in the upload → preview → mapping flow established by `TASK-0022`. The user has now requested that this mapping happen automatically for the known Broz column set, with target field names expressed as Czech-localized labels.

## Desired Outcome

- When the app detects a Broz CSV input, the column mapping step is pre-filled automatically with the following mapping (verbatim from the user's request):

  | Source column (English)    | Target field (Czech-localized)         |
  |-----------------------------|----------------------------------------|
  | `Timestamp`                 | `Časová značka zařízení`               |
  | `Record type`               | `Typ záznamu`                          |
  | `Glucose (mmol/L)`          | `Historie údajů o glukóze mmol/L`      |
  | `Scan glucose (mmol/L)`     | `Skenovat glukózu mmol/L`              |

- The user can still review and override the auto-mapping before confirming (consistent with the existing mapping UX from `TASK-0022`).
- The auto-mapping is robust to common variations in the source headers (case, whitespace, trailing units) within reason — exact tolerance to be defined in spec.
- The feature does not silently mis-map columns when the input is *not* a Broz CSV.

## Confirmed Facts

- The user provided the exact mapping table verbatim, reproduced above without modification.
- The target application "health-analysis" is registered in `applications/index.md` as **app-4 (Health-Analysis)**.
- Health-Analysis already supports Broz CSV ingestion at the loader level: per the `TASK-0030` implementation report, `loaders.py` has a content-detected double-quoted-line preprocessor for the Czech LibreView variant, and `samples/broz/Test1.csv` and `samples/broz/TEst.csv` exist as fixtures.
- `TASK-0022` established the upload → preview → mapping flow (typed frontend API client, mapping endpoint, in-memory upload store).
- `TASK-0030` confirmed Broz CSVs are valid input under the existing CGM dataset type and that "Czech-language column names are mapped exactly the same way LibreView columns are mapped via the existing column-mapping step" — i.e., the mapping infrastructure exists; this task is about *automating* the choice of mapping for Broz inputs.

## Assumptions (flagged, not confirmed)

- **Assumption:** The target field names listed in the user's mapping (`Časová značka zařízení`, `Typ záznamu`, `Historie údajů o glukóze mmol/L`, `Skenovat glukózu mmol/L`) are the Czech *display labels* that should appear in the mapping UI, not necessarily the canonical internal field IDs used by analytics. Internal canonical IDs likely remain stable English/snake_case identifiers (e.g. `device_timestamp`, `record_type`, `historic_glucose_mmol_l`, `scan_glucose_mmol_l`). **Needs confirmation.**
- **Assumption:** "Broz" is being used by the user as a label for a specific CSV export shape (likely a LibreView Czech variant). The exact device/vendor identity ("Broz" as a person/clinic name? a specific FreeStyle Libre export profile?) is **not confirmed** and should not be invented.
- **Likely:** Detection of a "Broz" CSV can reuse or extend the same content-detection logic introduced in `loaders.py` for the double-quoted-line preprocessor, since both share the same fixture set.
- **Likely:** The English source column names listed are the headers found inside the Broz CSV after the double-quoted-line unwrap performed by `loaders.py`.

## Constraints

- Treat as an app-4-only feature task.
- Must not weaken the explicit mapping/confirmation step in cases where the input is not confidently identified as Broz — silent automatic mapping of unknown inputs would be a clinical-data hazard.
- Must remain compatible with the deterministic-analytics-first safety framing.
- Must not regress existing LibreView ingestion (`TASK-0022`, `TASK-0023`, `TASK-0030`).
- Czech-localized labels must be preserved character-for-character (diacritics intact) in storage, display, and any normalization paths.

## Suspected Affected Areas

- `api/app/loaders.py` — Broz detection (possibly extending the existing content-detection logic).
- `api/app/...` mapping/preview endpoints (introduced in `TASK-0022`) — auto-fill of the mapping suggestion in the response.
- The canonical field-name registry / dataset-type schema on the backend (location to be confirmed) — to associate Czech display labels with canonical analytics field IDs.
- `web/lib/api.ts` and the upload/mapping page components — surfacing the auto-filled mapping with clear "auto-detected" affordances.
- Possible interaction with `TASK-0032` (i18n) if the Czech labels are part of a unified locale dictionary.

## Unknowns / Open Questions

- **What exactly is "Broz"?** A vendor? A device model? A clinic/user name? A specific export profile of FreeStyle Libre / LibreView? The user did not state this explicitly. Do not guess; flag for confirmation.
- Are the four columns above the *complete* Broz column set, or a representative subset? (`samples/broz/Test1.csv` may contain additional columns — e.g. device identifier, serial number — that need defined behavior.)
- Should the mapping be configurable/extensible (e.g., a JSON profile per source type) so future "X CSV → Y field" profiles can be added without code changes?
- How should missing columns be handled (Broz input lacking one of the four expected headers)?
- How should extra/unknown columns be handled (Broz input with additional columns beyond the four listed)?
- Are unit conversions implied by these mappings, or is `mmol/L` already canonical (per `TASK-0023` normalization, mmol/L is canonical, so likely no conversion is needed for these four)?
- Are the target field names display labels or canonical internal field IDs? (See Assumption above — strongly recommend display labels with stable internal IDs underneath.)
- How is "this is a Broz CSV" detected reliably without false positives against other LibreView-style inputs?
- If `TASK-0032` lands first, should the auto-mapped target labels be sourced from the i18n `cs` dictionary rather than hard-coded? (Recommended.)
