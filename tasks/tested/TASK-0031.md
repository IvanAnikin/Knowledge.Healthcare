# Task: Dynamic top-down loading and foldable Episodes table in Analysis window

- **Type:** feature

## Problem

The Analysis window (the per-analysis page in `Health-Analysis`) currently renders all of its content at once and includes an "Episodes" table that, for real-world CGM datasets, can grow to a very large number of rows. Two user-reported symptoms:

1. The Analysis window is slow to load / feels heavy because all content is rendered before the user sees anything useful at the top.
2. The Episodes table presents potentially thousands of records as a single scroll area, which is hard to navigate and visually overwhelms the rest of the analysis.

## Desired Outcome

- The Analysis window loads progressively, with the top-of-page content (summary, key metrics, primary charts) appearing and becoming interactive before later sections finish loading/rendering.
- Perceived load time of the most important top-of-page content is noticeably faster than today.
- The "Episodes" table is foldable/collapsible. By default it does not consume the full vertical space of the page; the user can expand it on demand.
- When expanded, the Episodes table remains usable for large datasets (no regression in the ability to inspect individual episodes).

## Confirmed Facts

- The user explicitly described two coupled requirements: (a) progressive top-down loading of the Analysis window, (b) foldable Episodes table to avoid scrolling thousands of rows.
- The target application is "health-analysis", which is registered in `applications/index.md` as **app-4 (Health-Analysis)**.
- The current Health-Analysis analysis page (`web/app/analysis/[id]/page.tsx` per `TASK-0025` and `TASK-0030` reports) already renders multiple statistics tables and 7 Plotly charts, plus the new monthly-comparison and recent-meals tables added in `TASK-0030`.

## Assumptions (flagged, not confirmed)

- **Likely:** The "Episodes" table refers to the deterministic CGM episode-detection output produced by the analytics core (`api/app/analytics/cgm/`) and rendered through `components/StatsTables.tsx` or similar on the analysis page. (Needs confirmation — there could be more than one table the user thinks of as "Episodes".)
- **Likely:** "Dynamically loaded from top" means progressive rendering / streamed or staged rendering of page sections (summary first, charts next, large tables last), not necessarily a backend pagination redesign.
- **Likely:** Current load behavior fetches the full `/api/v1/analyses/{id}` payload synchronously and renders all sections after the data arrives.

## Constraints

- App-4-only scope at this time.
- Must not regress the deterministic-analytics-first safety framing or hide clinically relevant information by default in a way that surprises clinicians.
- Must remain compatible with the existing `CgmAnalysisResult` contract and the `TASK-0030` additions (monthly comparison, recent-meal-events).
- Reduced-motion support and theme-aware rendering established by `TASK-0029` must continue to work.

## Suspected Affected Areas

- `web/app/analysis/[id]/page.tsx` — page composition and section ordering.
- `web/components/StatsTables.tsx` — likely home of the Episodes table component.
- Any chart components rendering Plotly figures — candidates for lazy-loading or deferred hydration.
- `web/lib/api.ts` — only if the data contract needs to be split for incremental loading (otherwise unchanged).
- Possibly `api/app/...` routes — only if a streaming or split endpoint is introduced; not required for a pure frontend progressive-render approach.

## Unknowns / Open Questions

- Exact pagination/virtualization strategy for the Episodes table: collapsed by default with full content on expand vs. virtualization (windowed list) vs. paginated vs. summary-only with "show all" affordance.
- What the default folded state should be (collapsed vs. expanded with N rows visible).
- Precise definition of an "episode" in this context — confirm it is the deterministic CGM episode output, not the meal-event/recent-meal table introduced in `TASK-0030`.
- Current real-world record volumes ("sometimes thousands") — actual maxima for representative datasets, to size virtualization/pagination correctly.
- Accessibility implications of the fold/expand pattern — keyboard, screen-reader, and reduced-motion behavior.
- Whether the "dynamic loading from top" requirement should be implemented as:
  - client-side progressive rendering (skeleton + staged hydration),
  - React Server Components / streaming SSR,
  - splitting the analysis API into multiple smaller endpoints, or
  - a combination.
- Whether print/export views (if any) should reflect the collapsed/expanded state or always render fully.

## Notes

- Related to `TASK-0029` (frontend redesign) and `TASK-0030` (added more tables to the analysis page), both of which already increased the vertical density of the analysis surface.
- This is primarily a UX/performance task; deterministic analytics outputs themselves are not changing.
