Use the following prompt verbatim when executing the implementation task in `/Users/ivananikin/Documents/Health-Analysis`:

```text
Create initial project documentation for a new web application for healthcare data analysis with an embedded LLM assistant.

Goal:
Build a high-level technical specification and implementation plan for a web application that allows users to upload healthcare datasets, run validated deterministic analytics, generate charts/statistics, and use an LLM layer to produce clinical-style summaries and interpretation support.

Important safety framing:
This application is clinical decision-support only. The LLM must not be framed as the final clinical decision maker. It should cite computed statistics, separate facts from interpretation, and phrase recommendations as “to be considered and verified by a clinician.” However, the long-term architecture may include an internal controlled agent mode that can generate analysis code, charts, and a draft conclusion after sandbox execution and review.

Please create the following documentation files:

1. `docs/PROJECT_OVERVIEW.md`
2. `docs/TECHNICAL_SPECIFICATION.md`
3. `docs/IMPLEMENTATION_PLAN.md`
4. `docs/ARCHITECTURE.md`
5. `docs/SAFETY_AND_GOVERNANCE.md`
6. `docs/ROADMAP.md`

Project concept:
The project should start with MVP Variant 1:

- Web app upload of CSV/XLSX files
- Dataset type selection:
  - continuous glucose monitoring / CGM time series
  - laboratory results
  - medication data
  - vital signs
  - diagnoses
  - generic time series
- Data preview and manual column mapping
- Schema validation
- Unit normalization
- Deterministic analytics modules
- Statistics and trends
- Chart generation
- LLM-generated summary based on computed results, not raw unvalidated data where avoidable

MVP reference pipeline:
Web application -> upload CSV/XLSX -> data validation -> deterministic analytical functions -> statistics/charts -> LLM summarizes results, explains findings, and proposes possible interpretations.

Backend architecture:
- Backend API
- Schema validation
- Unit normalization
- Deterministic analytical modules
- Computation of statistics and trends
- Chart generation
- Audit logging
- LLM orchestration

Frontend architecture:
- Upload CSV/XLSX
- Select dataset type
- Preview uploaded data
- Map columns
- Show validation errors and warnings
- Show computed statistics
- Show charts
- Show LLM commentary and clinical summary

LLM layer rules:
- Prefer passing computed statistics and structured results to the LLM instead of full raw data
- LLM must cite concrete computed values in its response
- Separate factual findings from interpretation
- Include uncertainty and data-quality warnings
- Phrase treatment-related suggestions as “to be considered by the clinician”
- Never present the LLM as having the final clinical authority
- Keep audit records of the prompt, model version, input summary, output, and analysis module version

Analytics MVP module:
Use continuous glucose monitoring / CGM as the first supported dataset type.

CGM analytics should include:
- LibreView-style CSV parsing
- Record type handling:
  - type 0 = regular CGM measurement every 15 minutes
  - type 1 = extra scan / patient-triggered measurement
  - type 6 = ignored/system record
- Time in ranges:
  - very low: <3.0 mmol/L
  - low: 3.0-3.8 mmol/L
  - target: 3.9-10.0 mmol/L
  - optimal: 3.9-7.8 mmol/L
  - high: 10.1-13.9 mmol/L
  - very high: >13.9 mmol/L
- Mean glucose
- Median glucose
- Standard deviation
- Coefficient of variation
- GMI
- Monthly backward-period analysis
- Daypart analysis:
  - night 00:00-06:00
  - morning 06:00-10:00
  - day 10:00-18:00
  - evening 18:00-24:00
- Hyperglycemic and hypoglycemic episode detection
- Rate-of-change analysis
- Approximate post-meal spike detection:
  - infer possible meal start from glycemic rise
  - assume meal likely occurred around 15 minutes before the start of the rise
  - analyze 3-hour post-meal windows
  - compare breakfast, lunch, dinner, and other periods
- Charts:
  - glucose time series
  - time in range bar chart
  - average daily/hourly profile
  - daypart comparison
  - monthly trend
  - heatmap day x hour
  - post-meal spike summary

Phase 2:
Internal analytical sandbox based on Copilot / VS Code / OpenCode workflow.

Purpose:
Allow developers or internal analysts to quickly create new deterministic analytical modules.

Requirements:
- Internal-only
- Uses local scripts and custom skills
- New analytics are first created as experimental modules
- They must be reviewed, tested, and promoted into production modules manually
- Output should include tests, assumptions, and documentation

Phase 3:
Controlled agent mode.

Workflow:
User describes the dataset -> agent proposes an analysis plan -> agent generates code -> code runs in an isolated container -> charts/statistics are generated -> agent writes a draft conclusion based on computed outputs.

Requirements:
- Isolated container
- No internet access by default
- No file-system access outside working directory
- Execution timeout and resource limits
- Code review or approval step before promotion to production
- Full audit log
- Clear separation between experimental agent-generated analysis and validated production analytics

Important architecture principles:
- Deterministic analytics first
- LLM only interprets validated outputs
- Human review for medical interpretation
- Modular analytics registry
- Reproducibility
- Auditability
- Data privacy
- Minimal raw patient data exposure to LLM
- Versioned analysis modules
- Clear validation and error reporting

Recommended stack:
Please propose a practical stack, preferably:
- Frontend: Next.js or React
- Backend: Python FastAPI
- Data processing: pandas, numpy
- Charts: Plotly or server-generated matplotlib/plotly exports
- Storage: local development file storage first, later object storage
- Database: PostgreSQL later for users, audit logs, analysis runs
- LLM: abstract provider interface, initially Azure OpenAI or OpenAI-compatible API
- Sandbox: Docker-based isolated runner in later phase

Please include:
- High-level architecture diagram in Mermaid
- Main modules and responsibilities
- Suggested folder structure
- Initial API endpoints
- Data model concepts
- Security and privacy considerations
- MVP acceptance criteria
- Implementation milestones
- Risks and mitigations
- Open questions

Tone:
Write the documentation in clear English, suitable for a technical founder/developer starting a new repository. Make it practical, implementation-oriented, and concise but complete enough to guide development.
```
