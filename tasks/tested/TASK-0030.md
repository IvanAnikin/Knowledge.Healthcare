## Problem

`Health-Analysis` already supports LibreView-style CGM ingestion and deterministic CGM analytics, but a new real-world customer-oriented use case has now been identified from sample files under `samples/broz/`.

The customer wants analysis that better matches what a doctor already sees in the LibreView system while also adding value beyond the default system reporting. The requested emphasis is on:

- month-by-month comparison tables of the core glucose metrics
- meal-rise / post-meal response analysis inferred from CGM rises
- per-meal tables for a recent window (for example the last 14 days)

This needs to be captured as a concrete app-4 task so the new data type / sample track and reporting expectations are visible in the roadmap.

## Desired Outcome

- `Health-Analysis` supports the `samples/broz/` CSV sample set as a first-class implementation/validation target within the CGM analytics track.
- The app can produce month-by-month comparison tables for the key CGM metrics side by side.
- The app can produce a post-meal analysis view based on inferred meal starts from glucose rises.
- The post-meal analysis includes, for each inferred meal/event:
  - glucose before meal
  - glucose after 1 hour
  - glucose after 2 hours
  - glucose after 3 hours
  - maximum glucose within the 3-hour window
  - time to maximum glucose
- The app can generate this meal-event table for a recent period such as the last 14 days.

### Key Design Decisions

- Monthly comparison reuses existing `monthly_trend` per-month `WindowStats` rather than introducing a new analytics primitive. The side-by-side experience is purely a presentation transform.
- Recent-meal events extend the existing TASK-0024 heuristic: `analyze_post_meal` still owns rise detection; the new function only enriches detected meals with pre-meal / +1h / +2h / +3h / max / time-to-max, then filters to the 14-day window. Avoids duplicating rise-detection logic.
- Sampling tolerance for offset readings is `±12 min`, just under the 15-min LibreView cadence, so a far-off reading is never reported across a sensor gap; missing readings are returned as `null`.
- `MODULE_VERSION` bump `1.0.0` → `1.1.0` because the result schema is extended.
- `recent_meal_events` has a default factory so external constructors of `CgmAnalysisResult` (e.g. `test_summaries.py`) keep working without churn.
- Heuristic framing is explicit: `RecentMealEvents.is_heuristic=True`, the table carries a heuristic badge with explanatory copy, and a new module-level assumption line states the offset-matching tolerance.

### Assumptions Recorded

- Broz CSVs are valid input under the existing CGM dataset type rather than a new dataset type. Czech-language column names are mapped exactly the same way LibreView columns are mapped via the existing column-mapping step.
- Recent-meal table defaults to 14 days; value held in `RECENT_MEALS_WINDOW_DAYS` constant rather than a request parameter — kept minimal until configuration is requested.
- "Pre-meal" = the latest reading at-or-before the inferred meal start (≈ `rise_start − 15 min`).
- "Time to max" is measured from the inferred meal start.
- Record type `5` (food/event marker rows in the Broz fixture) currently raises an `unknown_record_types` warning in validation but is excluded from analysis. The existing validator was intentionally left untouched.
