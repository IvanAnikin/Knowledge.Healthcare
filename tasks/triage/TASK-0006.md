## Deferral Note (2026-04-20)

This task is deferred to backlog. The investigation is not currently a priority. It may be revisited as part of a broader evaluation of whether ANOTE-web should consolidate to a single backend. Scope remains app-3 only — do not expand to cross-app work until this task is explicitly reactivated.


The demo page (`/demo`) renders slowly on first open after clicking the demo button. The delay is described as "several seconds." Subsequent opens may be faster. The cause is unknown.

## Desired Outcome

- The root cause of the slow first-load is identified and documented.
- If the fix is straightforward (e.g., lazy loading, bundle splitting, removing a blocking render), a follow-up implementation task is created.
- If the fix is complex or requires backend changes, it is documented as a separate task with appropriate priority.
- After any fix, the first open of the demo page should feel responsive (target: visible content within 1–2 seconds on a typical broadband connection).

## Constraints

- Investigation is limited to ANOTE-web (frontend and its API routes). Do not assume backend changes are required unless the investigation confirms it.
- The ANOTE FastAPI backend is a shared dependency (also used by App 2). Any backend changes would be cross-app and require separate coordination.
- No new infrastructure changes should be made without a separate task and sign-off.

## Suspected Affected Areas

- `src/app/[lang]/demo/page.tsx` — demo page component
- `src/components/demo/` — DemoUI and sub-components (may have heavy client-side initialization)
- `src/hooks/useDemoSession.ts` — 395-line state machine; may have expensive initialization
- `next.config.ts` — bundle optimization config (Framer Motion tree-shaking, etc.)
- `src/app/api/demo/transcribe/route.ts` and `src/app/api/demo/report/route.ts` — API routes that proxy to external services
- Azure SWA cold start behavior — serverless functions may have cold start latency on first request

## Confirmed Facts

- The demo page is at `src/app/[lang]/demo/page.tsx` (confirmed: `app-3-overview.md`).
- The demo uses `useDemoSession.ts` (395 lines, `useReducer`-based state machine) and `useMediaRecorder.ts` (confirmed: `app-3-overview.md`).
- DemoUI, RecordingControls, TranscriptPanel, ReportPanel, VisitTypeSelector are the demo sub-components (confirmed: `app-3-overview.md`).
- The demo calls Azure Whisper and the ANOTE backend via server-side proxy routes (confirmed: `app-3-overview.md`).
- Azure SWA API routes run as serverless functions that may cold-start (confirmed: `app-3-overview.md` deployment section).
- Framer Motion tree-shaking is configured in `next.config.ts` (confirmed: `app-3-overview.md`).
- Lottie (`lottie-react`) is a dependency and may be used in the demo (confirmed: `app-3-overview.md` tech stack).

## Assumptions

- Inferred: The most likely causes of first-load slowness are: (a) large JS bundle for the demo page not being code-split, (b) Lottie animation file loading blocking render, (c) Azure SWA serverless function cold start on the first API call, or (d) the demo page eagerly initializing expensive state/hooks on mount.
- Inferred: The "several seconds" delay likely refers to time-to-interactive or time-to-first-meaningful-content on the demo page, not necessarily a network error.

## Open Questions

1. Does the slowness occur before any user interaction (page load itself is slow) or only after clicking record/start (first API call is slow)?
2. Is the slowness consistent across browsers and devices, or only on certain configurations?
3. Has a Lighthouse or WebPageTest audit been run on the `/demo` route? The `IMPLEMENTATION_PLAN.md` targets Lighthouse Performance >= 95, but this has not been verified.
4. Is the Azure SWA serverless function cold start measurable? (Can be tested by calling `/api/demo/transcribe` directly with a minimal payload and measuring response time.)
5. Is Lottie used on the demo page? If so, what file size is the animation JSON?
6. Is the demo page statically rendered or server-rendered? (Next.js App Router behavior for dynamic routes needs to be confirmed.)

## Notes

- This is an investigation task. Its output is a diagnosis and a recommended fix, not a code change.
- If the investigation reveals a straightforward fix (e.g., add `next/dynamic` with `ssr: false` for a heavy component), that fix can be implemented in the same session and this task closed.
- If the fix requires significant refactoring or cross-app coordination, open a new task.
- Reference: `IMPLEMENTATION_PLAN.md` Lighthouse Performance target of >= 95 — the demo page may be pulling the score down.
