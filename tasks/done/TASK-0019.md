## Problem

The ANOTE brand logo needs to be changed consistently across the public web app and the mobile app.

Right now, the requested new logo is not yet propagated to the web favicon or to the mobile application icon surfaces. This creates branding inconsistency across products and leaves the currently installed/mobile-visible app icon out of sync with the intended ANOTE identity.

## Desired Outcome

- The ANOTE web app uses the new logo for its favicon.
- The ANOTE mobile app uses the new logo for the launcher/app icon shown on the home screen and other OS-managed app-icon surfaces.
- Android and iOS mobile icon sets are regenerated or replaced consistently from the approved source asset.
- Any required web icon metadata or supporting icon assets are updated so the new logo is what users see in the browser.
- The implementation is verified on the affected platforms/surfaces.

- The request explicitly spans more than one application: ANOTE-web (`app-3`) and ANOTE_mobile (`app-2`).
- ANOTE-web currently contains `src/app/favicon.ico`.

### Notification-bar icon (Android) — explicit decision

Per the task constraint to handle this explicitly rather than assume:

- Android 5.0+ requires status-bar/notification small icons to be **monochrome white silhouettes on a transparent background**; a colored launcher icon, when used as a notification icon, is rendered as a solid white square by the system.
- The current ANOTE logo is a colored wordmark on white. Producing a faithful monochrome silhouette is a brand/design decision (which mark element to silhouette, what stroke weight to preserve at small sizes), not a mechanical resize. Without an approved monochrome master, generating one here would be guessing the brand decision.
- The current codebase does not declare a custom notification small icon for the `flutter_foreground_task` ForegroundService in `AndroidManifest.xml`, so today the system falls back to plugin/default behavior; this is not made worse by this change.
- Recommendation: open a follow-up task to (a) get a monochrome master from the brand owner, (b) add it as `mipmap`/`drawable` assets (e.g. `ic_stat_anote.png` per density), and (c) wire it into the foreground-service notification config. Tracked in the handoff and current priorities for this session.

## Addendum (2026-04-27): Rounded Corners + Android Adaptive Icon

Follow-on visual polish requested by the product owner: bake rounded corners into web icons and upgrade Android to a proper adaptive icon. iOS was intentionally **not** modified — Apple disallows alpha and applies its own superellipse mask at runtime, so pre-rounding iOS icons is anti-pattern per Apple HIG.

### Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Corner radius | 22.5% of side (iOS-like) | Matches Apple superellipse approximation; balanced look on web. |
| Outside the rounded shape (web) | Transparent | Browsers/PWA install surfaces respect alpha. |
| iOS AppIcon | Skipped (left square) | Apple requires opaque RGB and applies its own mask. Pre-rounding is incorrect. |
| Android approach | Proper adaptive icon (with legacy rounded fallbacks) | Modern launchers mask shape per device; baked corners alone would conflict with the launcher mask on Android 8+. |
| Android adaptive background | `#FFFFFF` (matches existing wordmark background) | Consistent with current logo plate; can be revisited when a brand-approved colored plate is provided. |

### ANOTE-web changes

All under `src/app/`:

- `favicon.ico` — multi-size ICO (16/32/48/64) with 22.5% rounded corners and transparent outside.
- `icon.png` — 512x512 RGBA, rounded corners, transparent outside.
- `apple-icon.png` — 180x180 RGBA, rounded corners, transparent outside.
- `icon.svg` — wraps the existing wordmark paths in a `<clipPath>` rounded rectangle on a 424x424 viewBox.

Validation: `npm run build` clean. All four icon routes (`/favicon.ico`, `/icon.png`, `/icon.svg`, `/apple-icon.png`) registered as static.

### ANOTE_mobile Android changes

Adaptive icon now properly declared (didn't exist before this task):

- `mobile/android/app/src/main/res/mipmap-anydpi-v26/ic_launcher.xml` — adaptive-icon XML pointing at `@color/ic_launcher_background` and `@mipmap/ic_launcher_foreground`.
- `mobile/android/app/src/main/res/mipmap-anydpi-v26/ic_launcher_round.xml` — same adaptive icon for launchers that probe `ic_launcher_round`.
- `mobile/android/app/src/main/res/values/ic_launcher_background.xml` — declares the `ic_launcher_background` color resource (`#FFFFFF`).
- `mobile/android/app/src/main/res/mipmap-{mdpi,hdpi,xhdpi,xxhdpi,xxxhdpi}/ic_launcher_foreground.png` — full-bleed foreground PNGs at 108/162/216/324/432 px (the 108dp adaptive canvas convention). The wordmark sits inside the central 66dp safe zone (≈61.1% of side, centered) so it stays inside the launcher's mask whatever device shape is applied.

Legacy fallback (Android <8 only): the existing `mipmap-{mdpi,hdpi,xhdpi,xxhdpi,xxxhdpi}/ic_launcher.png` were re-baked with 22.5% rounded corners and transparent outside (overwriting the square versions produced earlier in TASK-0019). Modern Android ignores these in favor of the adaptive icon.

`AndroidManifest.xml` did not need to change — it already references `@mipmap/ic_launcher`, which now resolves to the adaptive icon on Android 8+ and to the rounded legacy PNG on older devices.

Pre-existing `values/colors.xml` was not present; no resource collisions with the new `values/ic_launcher_background.xml`.

Validation: `flutter analyze` reports the same 14 pre-existing `info`/`warning` issues (1 in `lib/services/whisper_service.dart`, 13 in `test/**`), all unrelated to icons. No new issues introduced.

### iOS

No changes. The iOS AppIcon set produced earlier in TASK-0019 (15 opaque PNGs, white background, no alpha) remains as-is. iOS will continue to apply its own superellipse mask at runtime on the home screen.
