## Problem

After the user stops a recording and the backend generates a report, pressing the **Start** button again does not correctly reset the app to a clean state. The current behavior leaves stale UI or functional state from the previous recording session, leading to inconsistent behavior on the second recording.

The expected behavior is that the following two user journeys should produce an **identical** UI and functional state before the new recording begins:

1. **Journey A (new session):** User presses "+" (new recording) → presses the record button to start.
2. **Journey B (re-record after stop):** User presses "Stop" → report is generated → user presses "Start" again.

Journey B currently does **not** fully match Journey A. The precise nature of the residual stale state (UI-only, functional, or both) is not yet confirmed from code inspection — see Unknowns below.

## Desired Outcome

- Pressing "Start" after a completed recording + report generation produces a state that is functionally and visually identical to starting a fresh recording via the "+" button.
- No stale transcript text, report content, recording metadata, audio buffers, or UI panel state carries over from the previous session.
- The fix must hold across all 6 supported visit types (default, initial, follow-up, gastroscopy, colonoscopy, ultrasound).
- All existing integration tests (`app_e2e_test.dart`, `recording_history_test.dart`) continue to pass.
- `flutter analyze` produces no new issues beyond the pre-existing 14-issue baseline.

## Constraints

- **No data loss:** The previous recording and its report must remain accessible in the local recording history (`recordings/_index.json`) after the reset — the reset applies only to active session state, not persisted history.
- **No backend changes required** unless the investigation reveals otherwise — this is expected to be a mobile-side state management fix.
- **GDPR by design:** In-memory audio buffers must be cleared on reset (they must never persist; this is already the intended design per `app-2-overview.md`).
- **Target device:** Samsung Galaxy S8 (Android 9, API 28) remains the primary validation target per the first customer setup.

## Suspected Affected Areas

- `mobile/lib/providers/session_provider.dart` — **Most likely root cause.** The Riverpod `StateNotifier` managing recording state. The `start new recording` action on "Start" may not be calling the same reset path as the "+" new-recording action.
- `mobile/lib/models/session_state.dart` — The `SessionState` model (recording status, session data fields). May need a `resetToInitial()` or equivalent that mirrors the clean state produced by "new recording."
- `mobile/lib/screens/home_screen.dart` — The main UI. The "Start" button handler after a completed session may differ from the "+" new-recording handler in what state it clears.
- `mobile/lib/widgets/` — Report panel, transcript panel, or recording panel widgets may hold local widget state that does not reset when the provider state changes.
- `mobile/lib/services/audio_service.dart` — Audio buffers; must be cleared/reset between sessions.
- `mobile/lib/services/whisper_service.dart` — On-device STT pipeline state; must be reset to avoid leftover VAD or model state.

## Unknowns

2. **Is there a single `reset()` / `clearSession()` method on the `SessionNotifier` that should be called by "Start", or does the fix require extracting a shared reset path?** — Unknown until code is inspected.
3. **Does the "+" button follow a different code path from the "Start" button post-recording?** — This appears to be the core issue, but the exact divergence point is unconfirmed.
4. **Are any widget-level local state variables involved (e.g., `setState()` in `home_screen.dart` or panel widgets)?** — Riverpod state reset may not automatically clear local `StatefulWidget` state.
5. **Is there a foreground-service or wakelock that needs to be properly stopped and re-initialized** between sessions? `flutter_foreground_task` and `wakelock_plus` are used; their teardown/setup cycle on re-record is unknown.
6. **Do any of the 17 existing E2E tests (26 March 2026 run) cover the Stop → report → Start re-record flow?** — If not, a new test case should be added as part of this fix.

## Resolution (2026-05-01)

**Root cause confirmed.** Journey B (post-report **Start**) went through `_saveAndClear(thenStartRecording: true)` which called `startNewRecording()` then `startRecording()` separately. `startNewRecording()` reset `state` via `state = SessionState();`, **clobbering `isModelLoaded` to `false`** (the default), and several private notifier fields (`_lastAutoSavedTranscript`, `_offlineFallbackModel`, `_cloudPartialIndex`) were never cleared between sessions. Result: Journey B ≠ Journey A.

**Fix.** In `mobile/lib/providers/session_provider.dart`:
- Strengthened `resetSession()`: clears the three private fields above and preserves `isModelLoaded` via `state = SessionState(isModelLoaded: _whisperService.isModelLoaded);`.
- Added `Future<void> restartRecording() async { await startNewRecording(); startRecording(); }` as the **single canonical Journey-B path**.
- Added defensive auto-reset at the top of `startRecording()` if state is dirty (transcript/report/visitTypeChanged/loadedRecordingId set).

In `mobile/lib/screens/home_screen.dart`:
- `_saveAndClear(thenStartRecording: true)` now calls `notifier.restartRecording()` (one canonical call) instead of separate `startNewRecording()` + `startRecording()`.

**Tests added.**
- `mobile/test/providers/session_provider_test.dart` (+3 tests): Journey A vs B `SessionState` equality; `resetSession` preserves `isModelLoaded`; `startRecording` auto-resets from dirty state. Also added platform-channel stubs (`flutter_secure_storage`, `wakelock_plus`) to fix pre-existing test breakage on those tests.
- `mobile/integration_test/journey_b_test.dart` (new, 239 LOC): device-grade end-to-end coverage of Journey B with fakes for audio/whisper/report/cloud services. Cannot execute in this session — no connected Android/iOS device.
