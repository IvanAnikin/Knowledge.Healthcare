## What Was Learned

- The Broz CSVs are a Czech LibreView variant where each row (including the header) is wrapped in an extra `"..."` layer. The implementation handles this with a content-detected `_maybe_unwrap_double_quoted_lines` preprocessor in `loaders.py`, gated on the header being fully wrapped and containing commas after unwrapping. Standard LibreView fixtures are unaffected.
- Monthly comparison did not require a new analytics primitive — it reuses `monthly_trend`'s per-month `WindowStats` and is implemented purely as a presentation transform (months as columns, metrics as rows).
- Recent-meal events extend the existing TASK-0024 heuristic without duplicating rise detection: `analyze_post_meal` still owns rise detection, and the new `compute_recent_meal_events` only enriches detected meals with pre-meal / +1h / +2h / +3h / max / time-to-max, then filters to the 14-day window.
- `RECENT_MEALS_WINDOW_DAYS=14` and `RECENT_MEALS_OFFSET_TOLERANCE_MINUTES=12` are constants, deliberately not yet exposed as request parameters. Tolerance is tuned just under the 15-min LibreView cadence so far-off readings are reported as `null` instead of being mismatched across sensor gaps.
- `CgmAnalysisResult` was extended with `recent_meal_events` carrying a default factory, so existing external constructors (e.g. `test_summaries.py`) continue to work without churn. `MODULE_VERSION` bumped `1.0.0` → `1.1.0` because the schema is extended.
- Heuristic framing is explicit on both sides: `RecentMealEvents.is_heuristic=True`, the table carries a heuristic badge with explanatory copy, and a new module-level assumption line states the offset-matching tolerance.
- Record type `5` (food/event marker rows in the Broz fixture) currently triggers an `unknown_record_types` validation warning and is excluded from analysis. The validator was intentionally left untouched per the "minimal changes" guidance; promoting record type `5` to `RECORD_TYPE_EXCLUDED_SILENT` is recorded as future work once LibreView semantics are confirmed.
- Playwright e2e visual snapshots under `web/e2e/analysis.spec.ts-snapshots/` are now stale because the analysis page renders two additional tables. They were not refreshed in this implementation session because the refresh requires a live server.
