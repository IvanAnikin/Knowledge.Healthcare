## What Was Learned

- Source `logo.PNG` is 424x424 RGBA but its alpha channel is fully opaque (extrema 255-255), so flattening to RGB for iOS marketing/App Store compliance does not lose any pixel detail.
- No ImageMagick / `rsvg-convert` / Inkscape was available on this machine, and `sips` cannot produce a multi-size `.ico`. Pillow (installed in a throwaway venv) was the simplest tool that could regenerate every required surface, including a proper multi-size ICO.
- Next.js App Router automatically picks up `favicon.ico`, `icon.png`, `icon.svg`, and `apple-icon.png` placed directly in `src/app/`. The `npm run build` log confirmed `/icon.png`, `/icon.svg`, and `/apple-icon.png` are now registered as static routes alongside the existing `/favicon.ico`.

## Decisions Made

- Used `ANOTE_mobile/logo.PNG` and `ANOTE_mobile/logo.svg` as the canonical source artwork. No new logo file was introduced.
- Did not touch any UI strings, layout, navigation, or non-icon brand surfaces.
- Did not flatten Android mipmaps to RGB — the launcher icon path tolerates RGBA, the artwork is effectively opaque, and changing pixel format unnecessarily would diverge from prior behavior.
- Did not generate an Android notification small icon: the colored wordmark cannot be turned into a faithful monochrome silhouette without a brand decision, and the current codebase does not declare a custom notification icon for the foreground service. This is recorded as an explicit follow-up rather than guessed.
- Did not introduce `flutter_launcher_icons` configuration. The existing folder structure was already complete; adding a tool would expand the surface without solving the actual logo-rollout requirement.
