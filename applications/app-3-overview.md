# Application 3 Overview — ANOTE Web (Marketing Website & Live Demo)

> Last updated: 2026-04-25. Based on direct inspection of the `ANOTE-web` repository plus user-reported production deployment notes for `TASK-0014`.

## Repository Name

`ANOTE-web` (GitHub: `IvanAnikin/ANOTE_web`) — local path: `/Users/ivananikin/Documents/ANOTE-web`

## Purpose

Marketing website and live browser-based demo for **ANOTE** — the AI-powered Czech medical dictation assistant. The site serves two functions:

1. **Marketing presence** — multi-page landing site (homepage, pricing, FAQ, report types, contact, legal pages) promoting the ANOTE product to prospective doctor customers.
2. **Live demo** — a fully functional in-browser demo at `/demo` where visitors can record voice or upload audio, see live transcription via Azure Whisper, and receive a structured Czech medical report generated by the ANOTE backend.

The website supports two locales: Czech (default) and English.

**Production URL:** `https://yellow-forest-086a45303.7.azurestaticapps.net` (Azure Static Web Apps, West Europe)

Source: `README.md:1-5`, `IMPLEMENTATION_PLAN.md:1-16`

## Users

- **Primary:** Prospective Czech-speaking doctors (internists, gastroenterologists) evaluating the ANOTE product
- **Secondary:** English-speaking visitors (international audience, investors, partners)
- **Admin:** Internal team member using `/admin` to view contact form submissions (protected by `ADMIN_SECRET`)
- **Access model:** Public anonymous access, no login required. Admin route uses a shared password.

Source: `README.md:1-3,37-43`, `IMPLEMENTATION_PLAN.md:10-16`

## Responsibilities

Within the broader healthcare ecosystem, this application is responsible for:

1. **Product marketing** — public-facing website with hero, features, how-it-works, pricing, testimonials, report type showcase, FAQ, and contact form
2. **Live demo gateway** — browser-based voice recording/upload with real-time transcription and report generation, proxied through server-side API routes (API keys stay server-side)
3. **Lead capture** — contact form submissions stored as JSON on disk and optionally emailed via SMTP
4. **SEO and i18n** — dual-locale (Czech/English) pages with JSON-LD structured data, sitemap, robots.txt, Open Graph metadata, and hreflang tags
5. **Analytics** — Plausible Analytics integration (cookie-free) with custom events for CTAs, demo usage, FAQ interactions, form submissions

This application does **not** handle: actual medical dictation workflows (that's App 2 — ANOTE Mobile), patient education (that's App 1 — Medical Advisor), persistent user accounts, or medical data storage.

## Data Handled

| Data Type | Sensitivity | Storage | Notes |
|-----------|-------------|---------|-------|
| Audio recordings (browser demo) | High (potential patient voice in demo misuse) | In-memory (browser) — never persisted server-side | Users can record up to 10 min or upload up to 25 MB. Rate limited. Demo disclaimer warns against real patient data. |
| Transcription text (demo) | Medium (demo context) | Transient — passed through to ANOTE backend, not stored | Server-side proxy forwards to Azure Whisper, returns result to client |
| Generated medical reports (demo) | Medium (demo context) | Transient — returned to client, not stored server-side | Client can copy/download. Not persisted. |
| Contact form submissions | Low (business lead info: name, email, phone, message) | File-based JSON (`data/submissions.json`) on the SWA runtime | Also emailed via SMTP if configured |
| Admin password | Secret | Azure App Settings / `.env.local` | `ADMIN_SECRET` env var |
| Azure Whisper API key | Secret | Azure App Settings / `.env.local` | Never exposed to client (server-side only, no `NEXT_PUBLIC_` prefix) |
| ANOTE API token | Secret | Azure App Settings / `.env.local` | Bearer token for backend, never exposed to client |
| SMTP credentials | Secret | Azure App Settings / `.env.local` | Gmail app password for email sending |
| Plausible analytics data | Low (pageviews, custom events) | Plausible Cloud | No cookies, no PII |

## Confirmed Facts

## Assumptions

These are reasonable inferences that should be verified:

1. **The West Europe migration has been completed (partially or fully).** The current live hostname `yellow-forest-086a45303.7.azurestaticapps.net` uses a `.7` suffix, which differs from the East US 2 hostname `ashy-sea-04ea79a0f.1.azurestaticapps.net` documented in the migration plan. This suggests a new SWA resource was created.
2. **The two GitHub Actions workflows may cause duplicate builds.** Both trigger on push to `main` with slightly different configurations. The manually authored one (`azure-static-web-apps.yml`) includes type checking; the auto-generated one does not.
3. **The `data/submissions.json` file-based storage is volatile on Azure SWA.** Azure Static Web Apps serverless functions run in ephemeral containers — file writes to `data/submissions.json` may not persist across function invocations. This is a potential data loss risk for contact form submissions.
4. **Some IMPLEMENTATION_PLAN.md items marked as incomplete are actually done.** Phase 4 (Live Browser Demo) is marked with `- [ ]` checkboxes but the demo is fully implemented in the codebase (`src/app/[lang]/demo/`, `src/components/demo/`, `src/hooks/useDemoSession.ts`). The plan appears to have been written before the demo was built and not fully updated afterward.
5. **Plausible Analytics may not be active.** The `.env.example` references `NEXT_PUBLIC_PLAUSIBLE_DOMAIN=anote.cz`, but whether a Plausible account is configured is not verifiable from the codebase.
